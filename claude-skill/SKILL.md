# 报销助手 Skill

将一堆发票图片/PDF 自动识别、按报销周期归档、分类、配对、整理，并为每个周期生成独立的 Excel 报销统计报表。

## 核心概念：报销周期

用户的发票往往来自多次出差，混在一起。用户会在输入目录中建几个**空文件夹**来标注报销周期，文件夹名包含时间信息。Skill 需要先识别这些周期文件夹，再把每张发票按消费日期归入对应周期，最后在每个周期内分别做分类和统计。

### 周期文件夹识别

扫描输入目录下的**空文件夹**（或只包含子文件夹、不直接包含发票文件的文件夹），从文件夹名中提取时间范围。常见命名格式：

| 文件夹名示例 | 解析结果 |
|-------------|---------|
| `0106-0108北京出差` | 周期：01-06 至 01-08 |
| `2024.01.06-01.08` | 周期：2024-01-06 至 2024-01-08 |
| `1月6日-8日出差` | 周期：01-06 至 01-08 |
| `0115上海` | 单日：01-15 |
| `北京出差0301-0305` | 周期：03-01 至 03-05 |

提取规则：
- 从文件夹名中找出所有日期片段（支持 MMDD、MM.DD、MM-DD、M月D日 等格式）
- 如果有两个日期，视为起止范围
- 如果只有一个日期，视为单日（该日期 ±0 天）
- 如果文件夹名包含年份则使用，否则根据发票中的年份推断
- 非空文件夹（已有发票文件的）不视为周期标记，跳过

### 发票归入周期

每张发票识别出 `service_date` 后，按以下逻辑归入周期：

1. 如果 `service_date` 落在某个周期的时间范围内（含边界 ±1 天容差） → 归入该周期
2. 如果落在多个周期范围内 → 归入最接近的周期
3. 如果不落在任何周期内 → 归入时间上最接近的周期，但标记提醒用户确认
4. 如果没有任何周期文件夹（用户没建） → 退回为单周期模式，所有发票统一处理

## 工作流程

```
扫描输入目录
  ↓
识别周期文件夹（空文件夹，从名称提取时间范围）
  ↓
逐一识别所有发票文件（Claude 视觉）
  ↓
按 service_date 将发票归入对应周期
  ↓
每个周期内：分类 → 配对 → 整理到目录 → 生成 Excel 报表
  ↓
汇总展示所有周期的报销情况
```

## 第一步：确认输入

Ask the user for:
1. **输入目录**: 发票文件所在的文件夹路径（支持 jpg/jpeg/png/bmp/webp/pdf）
2. **输出目录**: 整理结果的输出路径（默认在输入目录同级创建 `报销结果/`）
3. **操作模式**: 复制还是移动原始文件（默认复制，保留原文件）

然后扫描输入目录，列出发现的周期文件夹和散落的发票文件数量，让用户确认后再开始处理。

## 第二步：扫描和识别

1. 递归扫描输入目录中所有支持的文件（jpg/jpeg/png/bmp/webp/pdf），跳过空的周期标记文件夹
2. 逐一用 Read 工具查看每个文件（Claude 是多模态的，可以直接看图片和 PDF）
3. 对每个文件，提取以下信息并记录到一个 JSON 结构中：

```json
{
  "file_path": "原始文件路径",
  "type": "taxi|train|flight|hotel|meal|other",
  "subtype": "具体平台，如滴滴出行、12306、XX航空",
  "amount": 123.45,
  "date": "YYYY-MM-DD (开票日期)",
  "service_date": "YYYY-MM-DD (实际消费日期，比开票日期更重要)",
  "merchant": "商家名称",
  "invoice_number": "发票号码",
  "order_number": "订单号/交易号",
  "is_invoice": true,
  "description": "简要描述",
  "confidence": "high|low"
}
```

### 分类规则

| type | 类别名 | 识别范围 |
|------|--------|---------|
| taxi | 打车票 | 滴滴、高德、美团打车、曹操、首汽、出租车、网约车 |
| train | 火车飞机票 | 12306、火车票、高铁、动车 |
| flight | 火车飞机票 | 各航空公司、机票、登机牌 |
| hotel | 住宿费 | 酒店、宾馆、民宿 |
| meal | 餐费 | 餐厅、外卖、美团、饿了么 |
| other | 其他 | 无法识别的 |

### is_invoice 判断

- `true`: 正式发票（有发票代码、发票号码、税额等）
- `false`: 行程单、收据、凭证、水单等非正式发票

### 日期提取要点

- `service_date`（实际消费日期）比 `date`（开票日期）更重要
- 打车：乘车时间；火车/飞机：出发日期；酒店：入住日期；餐饮：消费日期
- 发票可能后补开，开票日期可能晚于消费日期

### confidence 判断（非常重要）

对每张发票的识别结果打一个置信度标记，决定是否自动处理还是交给用户确认：

**confidence = "high"**（可以自动处理）：
- 能清晰看到发票内容，金额/日期/商家都能确定
- 分类明确（比如明显是滴滴打车票、12306 火车票）
- service_date 能明确提取

**confidence = "low"**（必须放入「待人工确认」）：
- 图片模糊、看不清关键信息
- 金额无法确定或为 0
- 无法判断消费日期（service_date 提取不出来）
- 不确定是发票还是凭证
- 不确定属于哪个分类
- 文件内容不像发票（可能是截图、聊天记录、合同等杂项）
- 任何你拿不准的情况

**原则：宁可多放几张到「待人工确认」，也不要猜错了乱归类。用户手动处理几张比回头纠错省事得多。**

## 第三步：按周期归档

将识别完的发票按 `service_date` 分配到各个周期。

归入规则：
1. `service_date` 落在周期范围内（含边界 ±1 天容差） → 归入该周期
2. 落在多个周期范围内 → 归入最接近的
3. **`service_date` 不在任何周期内，或者 confidence="low" → 不要猜，放入「待人工确认」**
4. 没有周期文件夹 → 单周期模式，所有发票统一处理（low confidence 的仍然单独列出）

处理完后展示分配结果，**必须等用户确认后再继续**：

```
发现 3 个报销周期：
  [1] 0106-0108北京出差 (01-06 ~ 01-08): 12 张发票
  [2] 0115上海出差 (01-15): 5 张发票
  [3] 0201-0203深圳出差 (02-01 ~ 02-03): 8 张发票

⚠️ 以下 3 张需要你手动确认：
  1. 打车_0104.pdf — 日期 01-04 不在任何周期内，你想归到哪个周期？
  2. IMG_2831.jpg — 图片模糊看不清金额，疑似餐费发票
  3. scan_003.pdf — 不确定是发票还是合同

这些文件会放到「待人工确认」文件夹，你可以之后手动处理。
确认无误后我继续整理？
```

## 第四步：智能配对

在每个周期内，对发票和凭证做配对。打车、住宿等场景经常有「凭证/行程单 + 发票」两份文件：

配对规则（至少满足 2 个条件才配对）：
1. **订单号匹配**（最高优先级）：订单号完全相同 → 直接配对
2. **平台/商家匹配**：同一平台（如都是"滴滴出行"）
3. **日期匹配**：实际消费日期相同或相差 ±1 天
4. **金额匹配**：金额相同或差异在 ±5% 以内
5. **类型匹配**：同一 type

## 第五步：整理文件

将文件复制/移动到输出目录。每个周期一个顶层文件夹，内部按类别分：

```
报销结果/
├── 0106-0108北京出差/
│   ├── 打车票/
│   │   └── 2024-01-06_滴滴出行_35.00元/
│   │       ├── 01_2024-01-06_凭证_滴滴出行_35.00元.jpg
│   │       └── 02_2024-01-06_发票_滴滴出行_35.00元.pdf
│   ├── 火车飞机票/
│   ├── 住宿费/
│   ├── 餐费/
│   ├── 其他/
│   └── 报销统计.xlsx          ← 本周期的报表
│
├── 0115上海出差/
│   ├── 打车票/
│   ├── ...
│   └── 报销统计.xlsx
│
├── 待人工确认/                  ← 所有拿不准的都放这里
│   ├── 打车_0104.pdf           （保留原文件名，不重命名）
│   ├── IMG_2831.jpg
│   ├── scan_003.pdf
│   └── 确认清单.md             ← 说明每个文件的情况和疑问
│
└── 汇总报销统计.xlsx            ← 所有周期的总汇总（不含待确认）
```

**「待人工确认」文件夹规则：**
- 放在输出根目录（不归入任何周期）
- 保留原始文件名，不重命名（方便用户对照）
- 生成一份 `确认清单.md`，列出每个文件的情况：识别出了什么、哪里不确定、建议归到哪里
- 这些文件**不计入**任何周期的报表统计
- 用户确认后可以手动移到对应周期

文件夹命名：`{消费日期}_{商家}_{金额}元`
文件命名：`[序号_]{消费日期}_{发票|凭证}_{商家}_{金额}元.{ext}`

如果没有周期文件夹（单周期模式），则不生成周期层级，直接按类别组织。

## 第六步：生成 Excel 报表

对每个周期分别运行 `scripts/generate_report.py` 生成该周期的 Excel 报表。

用法：
```bash
python <skill-path>/scripts/generate_report.py <period_output_dir> <invoice_data_json_path>
```

其中 `invoice_data_json_path` 是一个 JSON 文件，包含该周期内所有已整理的发票信息数组。格式：
```json
[
  {
    "file_path": "整理后的文件路径",
    "type": "taxi",
    "category_name": "打车票",
    "amount": 35.00,
    "date": "2024-01-15",
    "service_date": "2024-01-15",
    "merchant": "滴滴出行",
    "subtype": "滴滴出行",
    "invoice_number": "12345678",
    "is_invoice": true,
    "description": "北京-朝阳"
  }
]
```

每个周期的报表包含：
- **汇总表**：各类别发票数量、金额小计、总金额
- **分类明细表**：每个类别一个 sheet，包含序号、日期、商家、金额、发票号码、描述

然后再生成一个**总汇总报表**（`汇总报销统计.xlsx`），放在输出根目录，汇总所有周期：
- 每个周期一行，显示周期名、发票总数、总金额
- 底部合计行

总汇总报表用脚本生成：
```bash
python <skill-path>/scripts/generate_report.py <output_root_dir> <all_periods_json_path> --summary
```

`--summary` 模式的 JSON 格式：
```json
[
  {
    "period_name": "0106-0108北京出差",
    "invoice_count": 10,
    "total_amount": 1580.50,
    "categories": {
      "打车票": {"count": 3, "amount": 185.50},
      "火车飞机票": {"count": 2, "amount": 526.00},
      "住宿费": {"count": 3, "amount": 645.00},
      "餐费": {"count": 2, "amount": 224.00}
    }
  }
]
```

## 重要注意事项

- **不确定就不要动**：任何拿不准的文件一律放「待人工确认」，附上说明，让用户自己决定。绝对不要猜测归类。
- 金额提取实际支付金额，不是税额
- 日期统一转换为 YYYY-MM-DD 格式
- 配对组的文件夹日期以凭证/行程单的实际消费日期为准
- 只有正式发票（is_invoice=true）且 confidence="high" 才计入报表金额
- 如果发票数量较多（>20），建议分批处理并给用户进度反馈
- 生成报表前先 `pip install openpyxl`（如未安装）
- 周期归档后**必须**展示分配结果等用户确认，不要自动继续
- 「待人工确认」文件夹里要生成 `确认清单.md`，格式如下：

```markdown
# 待人工确认清单

以下文件需要你手动确认后归入对应周期和分类。

## 1. 打车_0104.pdf
- **识别结果**: 疑似打车发票，¥35.00
- **问题**: 消费日期 01-04 不在任何报销周期内
- **建议**: 可能属于「0106-0108北京出差」，请确认

## 2. IMG_2831.jpg
- **识别结果**: 图片模糊，疑似餐费
- **问题**: 无法确认金额和日期
- **建议**: 请检查原件

## 3. scan_003.pdf
- **识别结果**: 看起来像合同而非发票
- **问题**: 不确定是否为报销凭证
- **建议**: 如果不是发票可以忽略
```
